SASS basics

Two flavors

There are *.sass and *.scss file types.

We focus on *.scss since it's a superset of CSS

Variables


$i-am-a-sass-variable: "I have value!";

// Begin with $ and assigned like CSS properties
$width: 5px;

.main {
    width: $width;
}

$small: 1px;
$medium: 3px;
$large: 7px;
$alist: $small, $medium, $large;

.main .small {
    width: nth($alist, 0);
}

Variables can be: strings, numbers, colors, lists, maps, booleans

@see SASS Variables

functions


// Asume our previously created list
@function first($list) {
    @return nth($list, 1);
}

@function last($list) {
   @return nth($list, length($list));
}

.main .large {
    width: last($alist)
}

@mixin h1-style($type: default) {
    font-family: Sans-serif;

    if ($type == sidebar ) {    // Branching logic in your CSS!

        line-height: 2;

@each $term in $terms {

    }

Compiling SASS

Toolkit

You need Ruby, and you need to install the sass gem


$ gem install sass

You can install Compass while you're at it


$ gem install compass

Basic compile

Give it an input SASS file and an output CSS file


$ sass input.scss output.css

Or give it an input and output directory


$ sass app/src/sass app/build

Background compile

Compile while you develop


$ sass --watch input.scss:output.css

This will monitor changes in your file and compile whenever you make a change to a file that is imported by your target file


$ sass --watch app/src/sass:app/src/build

Also works for watching directories

Project organization

The way it was

It used to be you had a file with 10K+ lines of CSS!

SASS lets you break down your CSS into mulitple files (partials) and @import them separately.

The way it could be

Think of your CSS code as modules

I like the Boostrap approach. Think: core, components, utilities

core your project's grid/layout, tables, buttons, forms

components the header, footer, frontpage, navigation bars

utilities auxiliary helpers like mixin libraries, variables

Typical SASS project

Define the compilation target


\Project
    \src
        \sass
            _mixins.scss
            _variables.scss
            styles.scss
    \build
        \styles
            styles.css

In this example we're defining the styles.scss to be our target and compiles to styles.css

In general, given a target source directory, SASS will never compile files that being with _ (partial files) into standalone CSS files

The target file


// styles.scss

@import "variables";
@import "mixins";

// Utilities

// Core

// Components

Your target file will usually import everything. Order matters. Variables and mixins are usually first.

Everything else


// styles.scss

@import "variables";
@import "mixins";

// Utilities
@import "responsive"
@import "utilities"

// Core
@import "tables"
@import "forms"

// Components
@import "header"
@import "footers"

You start importing an external framework (like Boostrap)

Compilation times get slower (up to 20x slower!)

Solution

Break up your files

    \src
        \sass
            _mixins.scss
            _variables.scss
            core.scss
            components.scss
            utitlities.scss
    \build
        \styles
            core.css
            components.css
            utitlities.css

Compilation times are faster (up to 20x faster!)

Downside is that you get duplicate code in the separate files

Lucky for us Moodle generates a single CSS file internally!

Nesting pitfalls

Nesting is very compelling the first time you start using SASS

Deep nesting creates very specific CSS styles -- hard to override later

Only nest up to three levels -- makes it easier to cascade

My approach

Use body classes as namespace


// header.scss
header {
    .header-logo { 
        width: 100px;
    }
    .header-login { .. }
    .header-navigation { .. }
}

    // Make logo bigger for front page

        .header-logo { 
            width: 200px;
        }
    }
}

Solution

Use SASS @extend inheritance. Use the cascade!


// progress-bars.scss

// Moodle's progress bar class inherits Bootstrap's progress bar styles
.dndupload-progress-outer {
    @extend .progress;
    @extend .progress-striped;
    @extend .active;
}
.dndupload-progress-inner {
    @extend .progress-bar;
}

@see Bootstrap Progress bars

Overriding CSS gracefully

@extend inheritance lets you override big project CSS gracefully

We use it to override default button styles, navigation tabs, forms -- This way we don't have to find every instance of a class and rename it. We simply override it using inheritance.

Separation of structure and presentation

@extend vs @mixin

These look very similar, and can solve the same problems. However, they work very differently

@extend vs @mixin

@extend does not copy styles, it copies the selector -- good for smaller CSS files


// somewhere in your CSS
.dndupload-progress-outer,
.progress {
...
}
.dndupload-progress-outer,
.progress-striped {
...
}
.dndupload-progress-outer,
.active {
...
}

@mixin will copy code.

@extend vs @mixin

@mixin can have parameters and branching logic

@extend is meant to solve different problems.

@extend limitation

@extend cannot, uh, extend a nested selector.


// somewhere in your CSS
.progress-bar {
    /// defined
    .active {
        /// defined
    }
}

// Not possible.
.dndupload-progress-inner {
    @extend .progress-bar .active;
}

What just happened?

We used a trait-pattern to extend the functionality of buttons

We used an adapter-pattern using our place holder definitions

Since we're using @extend we did it without copying (much) CSS code - copied selectors instead.

Cool sass trick

@content

Pass content blocks to mixins

Problem?

Responsive design is hard. Suppose you want a responsive header


// header.scss
header {
    .header-logo { 
        width: 100px;
    }
}

// Define a @media query
// Make logo smaller and center it
@media (min-width: $size) { 
    header {
        .header-logo { 
            width: 50px;
            text-align: center;
        }
    }
}

The SASS way


// header.scss
header {
    .header-logo { 
        width: 100px;
        
        // Include your responsive layout change inline
        @include respond-to(phone) {
            width: 50px;
            text-align: center;
        }
    }
}

The @mixin

The @mixin's content block is copied to @content


// mixins.scss
@mixin respond-to($screen) {
    if ($screen == phone) {
        @media (min-width: $size) { 
            @content;   // Magic!
        }
    else if ($screen == ...) { .. }
}

Bootstrap

Possible to import Boostrap-sass project via Compass

You'll need config.rb file


## Import boostrap package by adding path to package source
additional_import_paths = ["vendor/twbs/bootstrap-sass/assets/stylesheets"]


// buttons.scss

// Import the bootstrap buttons source
@import "bootstrap/buttons";

// Modify to make buttons look squared
.btn {
    @include border-radius(0);
}

// Make form submit buttons look like boostrap primary buttons
form {
    input[type="submit"] {
        @extend .btn;
        @extend .btn-primary;
    }
}
